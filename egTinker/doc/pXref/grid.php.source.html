   6  /**
   7   *  Grid access class
   8   *
   9   *  This class allows you to connect to a remote Grid UI server and
  10   *  launch and monitor jobs on it.
  11   *
  12   *  The reason for this class is mostly one of resilience: if you 
  13   *  put all your services directly on a GrUI host, then whenever 
  14   *  that host if offline, your services will be as well.
  15   *
  16   *  An alternative is to replicate the services on various GrUI nodes.
  17   *
  18   *  Or better yet: use this class and set up your services wherever
  19   *  you want. You may put them on an HA system and ensure this way 
  20   *  their continuous availability. Your service will be always up
  21   *  and running, ready to accept jobs.
  22   *
  23   *  Your next problem is dealing with the Grid UI. You still need
  24   *  to log-in on a Grid access point to submit your jobs. But once
  25   *  you have detached from a specific GrUI node, you are free to
  26   *  attempt a connection to a given remote access point, and if it
  27   *  is available (and while it is) submit jobs as needed. If at any
  28   *  time it is not available, nothing is lost: just look for another
  29   *  one and use this to continue working. You may thus enter the
  30   *  grid through any door.
  31   *
  32   *  There is a side advantage too: with your services on a GrUI node
  33   *  you can only launch jobs from it. With your services detached from
  34   *  any given Grid door, you may use _any_ AND _as many_ as you want:
  35   *  this means you may launch jobs using various GrUI nodes simultaneously
  36   *  if you so wish. And even split the jobs througho various, separate 
  37   *  Grids if you feel like it, hence potentially increasing your 
  38   *  throughput and computing power, harnessing even more resources.
  39   *
  40   *  Detaching your jobs from the GrUI has one serious drawback though:
  41   *  all your job information must travel from your service user interface
  42   *  to the grid user interface through the Internet, which may be
  43   *  potentially dangerous. We deal with this security issue using SSH
  44   *  to handle all communications and provide encryption. As long as 
  45   *  you use strong passwords you may feel secure. Actually, it is as
  46   *  weak (or strong) as working directly on the GrUI node: access to
  47   *  it is still managed by standard password mechanisms and subject to
  48   *  the same types of attacks. However, remember you now have an extra
  49   *  system (the front-end) to maintain and secure. Please, be always
  50   *  cautious with any server you use.
  51   *
  52   *  Sounds convincing? Then read ahead to learn how to use this class.
  53   *
  54   *  Using this class you may access the Grid in two different ways:
  55   *  connected or disconnected mode (or if you prefer, using persistent
  56   *  connections or disconnected mode).
  57   *
  58   *  It is important to understand the differences among them:
  59   *
  60   *  PERSISTENT CONNECTIONS
  61   *
  62   *  When working with persistent connections, you first issue a
  63   *  Grid::connect() call to stablish a connection to the GrUI. This will
  64   *  be kept open during all the time until you call Grid::disconnect().
  65   *  All commands issued will travel over the same wire, and hence 
  66   *  communications will be more efficient.
  67   *
  68   *  Output of all your commands is collected on a single I/O channel
  69   *  that persists during the whole session.
  70   *
  71   *  On the minus side, persistent connections drive all communications
  72   *  between your front-end and the grid access point through pipes, and
  73   *  are subject to serious deadlock problems. You must make sure you
  74   *  avoid them by clearing periodically the communication buffers. In
  75   *  addition, error checking is relayed to YOU: all information will 
  76   *  travel through the persistent I/O channels, and it will be YOUR
  77   *  responsability to detect errors and act accordingly. Debugging is
  78   *  thus more difficult.
  79   *
  80   *  DISCONNECTED MODE
  81   *
  82   *  When working in disconnected mode, the connections are open and
  83   *  closed for each command you issue. This imposes a heavier tax
  84   *  on your communications, which may become serious for submitting
  85   *  huge numbers of jobs.
  86   *
  87   *  Each command will return its output separately since it uses a
  88   *  new I/O channel every time.
  89   *
  90   *  On the plus side, you will retrieve job completion status inmediately
  91   *  and won't risk deadlocks while running remote jobs. Development and
  92   *  debugging will be a lot easier.
  93   *
  94   *  JOB MANAGEMENT
  95   *
  96   *  In order to submit your jobs to the grid you need to understand how
  97   *  job management has been defined for this class. On the command line
  98   *  you would have a lot more versatility, but to make this class more
  99   *  useful some compromises had to be reached. We have defined a strict
 100   *  protocol to generate/prepare your jobs before submitting them to
 101   *  the grid, and you must stick to it if you want to avoid problems.
 102   *
 103   *  To understand why this has been designed the way it is, you should
 104   *  keep in mind that you will be preparing your jobs on one (or many)
 105   *  front-end and submitting them from it to one (or many) GrUI nodes.
 106   *  Further to it, this has been designhed to make it easy to deploy
 107   *  web-based services for users. Therefore many similar jobs might
 108   *  be launched simultaneously and we need some way to avoid collusion
 109   *  among them. To avoid one job stepping over other we must isolate
 110   *  every one from each other. This means providing an isolated environment
 111   *  for every job.
 112   *
 113   *  The easiest way to achieve our goal is to have every job submitted
 114   *  placed on an independent directory (which we pair to the job name). 
 115   *  For single jobs, this means that you should make sure that no two 
 116   *  potentially simultaneous/overlapping jobs have the same name (i.e. 
 117   *  are stored in the same directory so there is no risk one overwrites
 118   *  files of the other).
 119   *
 120   *  Sometimes this may result inconvenient to you. E.g. if your whole job
 121   *  is submitted split into many separate sub-jobs (which each is a separate
 122   *  job from the point of view of the grid) you may want to follow some
 123   *  naming convention for your sub-jobs that makes it easier to identify
 124   *  and keep track of them. In this case, if you had two simultaneous
 125   *  runs, then the names would collide.
 126   *
 127   *  For example, let's say you are rendering frames of a movie and want
 128   *  to identify each job by frame number: 0000, 0001, 0002, 0003... If
 129   *  you now try to generate a second movie while the first is being
 130   *  processed, then the frames of the second movie (named as well 0000,
 131   *  0001, etc...) would overwrite the frames of the first one.
 132   *  Generating random names for each frame would be an option, but too
 133   *  cumbersome and expensive as you would need to keep track of the
 134   *  association of the random names with the actual frames.
 135   *
 136   *  To deal with this scenario we define 'sessions'. A session is identified
 137   *  by a unique identifier, and guarantees that all jobs belonging to this
 138   *  session are kept separate from similarly named jobs from other sessions.
 139   *
 140   *  Actually when you create a session, what we actually do is create
 141   *  a subdirectory in the GrUI and direct all further jobs to this 
 142   *  subdirectory. This way, jobs of two sessions may have the same name
 143   *  and not step into each other.
 144   *
 145   *  PREPARING JOBS FOR THE GRID
 146   *
 147   *  To prepare a job for the grid you must assign it a name. The same
 148   *  preacutions that apply to any local job hold for your grid work too:
 149   *  if various simultaneous jobs of the same kind may be run, then each
 150   *  must be kept separate from the others by using a different name.
 151   *
 152   *  Once you have decided the name, you must create a directory locally
 153   *  with the same name as your job. In this directory you must install
 154   *  everything needed to run your job: executables, libraries, input
 155   *  data and a JDL file.
 156   *
 157   *  The JDL file defines the work that we will ask the grid to carry
 158   *  out. Since each single job gets its own directory, you will only
 159   *  submit one JDL file for each, and to make processing easier, we
 160   *  request that this JDL file have a fixed name: "job.jdl".
 161   *
 162   *  The grid processing will generate various auxiliary files, for internal
 163   *  housekeeping. Again, for simplicity, we have chosen to call each of
 164   *  them 'job.*', i.e. 'job.' something. This means that other than 'job.jdl'
 165   *  you should NOT create any file named job.anything on your job directory
 166   *  to avoid collusion with possible temporary files.
 167   *
 168   *  In brief, to prepare a job:
 169   *  - select a name
 170   *  - create a directory named after the job
 171   *  - populate this directory with all files needed to run your job
 172   *  - generate the file 'job.jdl' with the description of the work to
 173   *    be carried out by the grid
 174   *  - avoid having files named 'job.*' (starting with 'job.')
 175   *
 176   *  SUBMITTING JOBS TO THE GRID
 177   *
 178   *  First consider whether you will be using unique job names or if
 179   *  you will follow a convenient naming convention that may cause name
 180   *  collisions with other jobs.
 181   *
 182   *  If you feel pretty safe that the job name is unique (e.g. has been
 183   *  generated using one or more random strings), then simply call the
 184   *  appropriate *job_submit() function.
 185   *
 186   *  If you are using names that have low entropy or reusing names for
 187   *  similar jobs then it is advisable that you first call one of the
 188   *  *new_session() routines to ensure all your jobs will be kept isolated
 189   *  from other similarly named jobs, and then use the *job_submit()
 190   *  routines to send your jobs.
 191   *
 192   *  For the curious: when you submit your job, the directory and all
 193   *  of its contents will be sent to the remote Grid UI selected and
 194   *  then the 'job.jdl' will be submitted to the grid. In the process,
 195   *  several files will be generated holding information about your
 196   *  job identity in the grid context that will be kept for housekeeping
 197   *  and future reference.
 198   *
 199   *  The above is to be kept in mind when submitting light or numerous
 200   *  jobs: the transfer time may become sensibly relevant. Please do 
 201   *  take it into consideration in your equations when designing jobs
 202   *  for the grid using this class. You may find it interesting to first
 203   *  store all or some of your job data/execs on the grid and keep them
 204   *  already there instead of having to copy them.
 205   *
 206   *  Grid file management routines are not included yet, but are intended
 207   *  for a future release of this class.
 208   *
 209   */

 335      /**
 336       * open a persistent connection to the Grid UI server
 337       * 
 338       *     The Grid User Interface Server is the entry point to the Grid
 339       *  for users and user applications. This is where jobs are launched from.
 340       * 
 341       *     This package has been designed to be able to be installed in any
 342       *  host, independent of whether it is an UI or not. Thus, to be able to
 343       *  submit jobs to the Grid, the server hosting the Web UI must connect to
 344       *  a Grid UI host to do the work.
 345       * 
 346       *     This routine opens a connection to a Grid UI host using an specified
 347       *  username (i.e. all jobs will be run under said username). To allow the
 348       *  caller to communicate with the remote end, it creates two pipes/pty, one for
 349       *  input and one for output, and redirects error messages to a file.
 350       *
 351       *  We need to redirect stderr to a file. This is so to avoid blocking on
 352       *  reading to check for errors and to avoid (if we use PTYs) interleave
 353       *    of error messages and normal I/O.
 354       * 
 355       *     These pipes lead to a child process that actually manages the 
 356       *  communication. Using a child process has several advantages: it simplifies
 357       *  the communication process by offloading the comm. logic to a separate,
 358       *  convenience tool, and by being able to use SSH as the child, we can increase
 359       *  security taking advantage of its encryption capabilities.
 360       * 
 361       *     The panorama therefore will look like this:
 362       * 
 363       *     HTML front-end --> processor.php <--> SSH <--> remote host <--> Grid
 364       *
 365       *    This allows for better resilience: should a GridUI host be 
 366       * unavailable, we can detect the error condition and try another 
 367       * one. If the GridUI runs the front-end, then we have a single
 368       * point of failure, which is a no-no.
 369       *
 370       *    @note    Use of persistent connections is greatly DISCOURAGED:
 371       * all I/O will go through pipes, and any end of the line may hang
 372       * waiting on read (if there is nothing to be read at the other end)
 373       * or write (if there is noone to retrieve the data at the other
 374       * end).
 375       *
 376       *    Just picture this: you open a connection to a GridUI and the
 377       * remote host issues an unusually large 'motd' that fills the
 378       * pipe. The remote shell will hang waiting for someone to read and
 379       * empty the pipe before continuing. Now, on the local side we do
 380       * issue a remote command (without checking the output): we hang 
 381       * waiting for the other end to read it, but the other end is hung.
 382       * Sadly a 'motd' may contain anything, and we can't predict what
 383       * the remote prompt will look like...
 384       *
 385       *    Be careful. Be _very_ careful.
 386       *
 387       *    AND always consider setting the pipe ends to non-blocking status.
 388       * This is actually the default, but has its tricks too. Be careful.
 389       * Be _very_ careful.
 390       *
 391       *    @note    There is no easy way to know the exit status of a remote
 392       * command while using persisten connections. Your only chance is to
 393       * review the output log and check it yourself for error messages.
 394       *
 395       *    We could be doing better here, but for now we will leave this for
 396       * later. After all, the point here is to give the user maximum
 397       * efficiency.
 398       *
 399       *    @return TRUE on success, FALSE otherwise.
 400       */

 402      function pconnect()
 403      {    
 404          global $php_version;
 405      global $debug;
 406      
 407      // Open a child process with the 'proc_open' function. 
 408      //
 409      // Some tricks: we must open the connection using '-x' to disable
 410      // X11 forwarding, and use '-t -t' to avoid SSH generating an error
 411      // because we are not connected to any terminal.
 412          // NOTE:
 413      //   We require users to have an account and password on
 414      //   the UI and provide their user/password through the web or
 415      //   otherwise (e.g. using myproxy)
 416          //
 417      // NOTE: if the web server is trusted remotely (i.e. it's SSH public 
 418      // key is accepted in ~user@host:.ssh/authorized_keys) then any 
 419      // password will do.
 420      if ($php_version < 5) {
 421          $descriptorspec = array(
 422              0 => array("pipe", "r"),  // connect child's stdin to the read end of a pipe
 423              1 => array("pipe", "w"),  // connect child's stdout to the write end of a pipe
 424              2 => array("file", $this->error_log, "a") // stderr is a file to write to
 425          );
 426          $this->process = proc_open("SSH/SSH.sh $this->entry_point", 
 427                       $descriptorspec,
 428                   $pipes);
 429          if ($debug) echo "SSH/SSH.sh $this->entry_point<br />\n";
 430          // check status
 431          if (!is_resource($this->process)) 
 432          {
 433          letal("Grid::connect", "cannot connect to the Grid");
 434          return $this->connected = FALSE;
 435          }
 436          if ($debug) echo "proc_open<br />\n";
 437          // give SSH it's due password
 438          // XXX - this might be better..
 439          fwrite($pipes[0], "$this->password\n");
 440          if ($debug) echo "$this->password<br />\n";
 441          fflush($pipes[0]);
 442       
 443      }
 444      else {     /* php5 -- we can use PTYs */
 445          /* XXX -- untested -- probably unneeded */
 446          $descriptorspec = array(
 447          0 => array("pty"),
 448          1 => array("pty"),
 449          2 => array("file", $this->error_log, "a")
 450          );
 451          $this->process = proc_open("ssh -x -t -t $this->entry_point", 
 452                           $descriptorspec,
 453                   $pipes);
 454          
 455          // check status
 456          if (!is_resource($this->process)) 
 457          {
 458          letal("Grid::connect", "cannot connect to the Grid");
 459          return $this->connected = FALSE;
 460          }
 461  
 462          $status = proc_get_status($this->process);
 463          if ($status->running == FALSE) {
 464          fclose($pipes[0]); fclose($pipes[1]); fclose($pipes[2]);
 465          proc_close($this->process);
 466          letal("Grid::connect", "connection exited ".$status->exitcode);
 467          return $this->connected = FALSE;
 468          }
 469          if ($status->signaled) {
 470          fclose($pipes[0]); fclose($pipes[1]); fclose($pipes[2]);
 471          proc_close($this->process);
 472          letal("Grid::connect", "connection terminated by ".$status->termsig);
 473          return $this->connected = FALSE;
 474          }
 475          if ($status->stopped) {
 476          // Tell the user and hope for the best
 477          warning("Grid::connect stopped by ".$status->stopsig.
 478          " it may still have a chance though");
 479          }
 480  
 481          // Give ssh it's due password
 482          $dummy = fgets($pipes[1],1024);    /* wait for prompt */
 483          if ($debug) echo $dummy."<br />\n";
 484          fwrite($pipes[0], "$this->password\n");
 485          fflush($pipes[0]);
 486  
 487      }
 488      
 489      // $pipes now looks like this:
 490      //   0 => writeable handle connected to child stdin
 491      //   1 => readable handle connected to child stdout
 492      // Any error output will be appended to $wd_path/error-output.txt
 493          $this->std_in = $pipes[0];
 494      $this->std_out = $pipes[1];
 495      $this->std_err = fopen($this->error_log, "r");
 496  
 497      // We now have a connection to the remote Grid User Interface
 498      // Server which we may use to send commands/receive output
 499      return $this->connected = TRUE;
 500      }

 502      /**
 503       * Close the connection with the remote Grid access point (UI node)
 504       *
 505       *     What we are going to do is close the communication pipes
 506       *  and kill the child process that actually handles communication
 507       *  with the remote Grid UI host (ssh).
 508       *  This function returns the exit status of the connection (i.e. of
 509       *    the intermediate program that handles the connection --in this
 510       *  case SSH).
 511       *
 512       *  @return integer exit status of the connection (or the handling program)
 513       */
 514      function pdisconnect()
 515      {
 516      global $debug;
 517      if ($this->connected == FALSE)
 518          return $ret = 0;
 519  
 520      // We need to close the 'child' process, otherwise the call to
 521      // the 'proc_close()' function hangs!!!
 522      fwrite($this->std_in, "logout\n");
 523      fflush($this->std_in);
 524  
 525      fclose($this->std_in);
 526  
 527      // The output from the commands
 528      //  Note that this may
 529      //      a) disrupt normal output
 530      //      b) get into an infinite loop if the child is in one
 531      //  But it is useful for debugging, hence here.
 532          if ($debug == TRUE)
 533          do {
 534              
 535          $line = fgets($this->std_out, 1024);
 536          echo $line."<br />\n";
 537          } while (!feof($this->std_out) && (strlen($line) != 0));
 538  
 539      fclose($this->std_out);
 540  
 541          if ($debug == TRUE)
 542              do {
 543              
 544          $line = fgets($this->std_err, 1024);
 545          echo $line."<br />\n";
 546          } while (!feof($this->std_err) && (strlen($line) != 0));
 547  
 548      fclose($this->std_err);
 549  
 550      // It's important that we close any open files before calling
 551      // proc_close in order to avoid a deadlock
 552  
 553      $return_value = proc_close($this->process);
 554      
 555      $this->connected = FALSE;
 556  
 557      return $return_value;    
 558      }

 562      /**
 563       * Start the Grid services
 564       *
 565       *    This function starts the grid services on the remote UI server host.
 566       *    This is done by unlocking the certificate that we are going to use 
 567       *  to run our jobs on the grid using the passphrase provided.
 568       *
 569       *    We need to have a connection open with the remote grid server as the 
 570       *  user with whose identity we want to run the jobs. This connection is
 571       *    created by Grid::pconnect().
 572       *
 573       *    To ease things up, we check if we are already connected, and if we
 574       *    aren't, we try to connect ourselves. That is, there is no need to
 575       *    call Grid::Connect() first unless you want to do something else
 576       *    on the Grid-UI before initializing the Grid.
 577       *
 578       *    Grid services have a lifetime of their own. By default they are
 579       *    available for 12:00 hours (that's the default value of
 580       *    grid-proxy-init itself), but their duration may be fine tuned
 581       *    if we have some knowledge about the time required to run our
 582       *    job.
 583       *
 584       *    Session duration is specified in hours+minutes. If the number of
 585       *    minutes is negative, the specified minutes are substracted from
 586       *    the specified hours (e.g: 1, -15 is fifteen minutes to one hour,
 587       *    i.e. 45 minutes). If the total time specified is negative then
 588       *    the default of 12:00 is used.
 589       *
 590       *    @param    hours        Estimated duration in hours of the session
 591       *

 776      /**
 777       * Start the Grid services
 778       *
 779       *    This function starts the grid services on the remote UI server host.
 780       *    This is done by unlocking the certificate that we are going to use 
 781       *  to run our jobs on the grid using the passphrase provided.
 782       *
 783       *    Grid services have a lifetime of their own. By default they are
 784       *    available for 12:00 hours (that's the default value of
 785       *    grid-proxy-init itself), but their duration may be fine tuned
 786       *    if we have some knowledge about the time required to run our
 787       *    job.
 788       *
 789       *    Session duration is specified in hours+minutes. If the number of
 790       *    minutes is negative, the specified minutes are substracted from
 791       *    the specified hours (e.g: 1, -15 is fifteen minutes to one hour,
 792       *    i.e. 45 minutes). If the total time specified is negative then
 793       *    the default of 12:00 is used.
 794       *
 795       *    @param    hours        Estimated duration in hours of the session
 796       *
 797       *    @param    minutes     Estimated duration in minutes of the session
 798       *
 799       *    @return TRUE on success, FALSE otherwise
 800       */
 801       
 802      function initialize($hours=12, $minutes=0)
 803      {
 804          $total_minutes = ($hours * 60) + ($minutes);
 805      if ($total_minutes < 0) {
 806          //use default
 807          $hours = 12;
 808          $minutes = 0;
 809      } else {
 810          $hours = floor($total_minutes / 60);
 811          $minutes = $total_minutes % 60);
 812      }
 813      
 814          $remote = SExec::ssh_popen("grid-proxy-init -pwstdin -valid $hours:$minutes", "w");
 815      fputs($remote, $this->passphrase);
 816      $status SExec::ssh_close($remote);
 817      if ($status == 0) {
 818          return $this->initialized = TRUE;
 819      } else {
 820          return $this->initialized = FALSE;
 821      }
 822      }
