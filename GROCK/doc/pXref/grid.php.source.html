  42  /**
  43   *  Grid access class
  44   *
  45   *  This class allows you to connect to a remote Grid UI server and
  46   *  launch and monitor jobs on it.
  47   *
  48   *  The reason for this class is mostly one of resilience: if you 
  49   *  put all your services directly on a GrUI host, then whenever 
  50   *  that host if offline, your services will be as well.
  51   *
  52   *  An alternative is to replicate the services on various GrUI nodes.
  53   *
  54   *  Or better yet: use this class and set up your services wherever
  55   *  you want. You may put them on an HA system and ensure this way 
  56   *  their continuous availability. Your service will be always up
  57   *  and running, ready to accept jobs.
  58   *
  59   *  Your next problem is dealing with the Grid UI. You still need
  60   *  to log-in on a Grid access point to submit your jobs. But once
  61   *  you have detached from a specific GrUI node, you are free to
  62   *  attempt a connection to a given remote access point, and if it
  63   *  is available (and while it is) submit jobs as needed. If at any
  64   *  time it is not available, nothing is lost: just look for another
  65   *  one and use this to continue working. You may thus enter the
  66   *  grid through any door.
  67   *
  68   *  There is a side advantage too: with your services on a GrUI node
  69   *  you can only launch jobs from it. With your services detached from
  70   *  any given Grid door, you may use _any_ AND _as many_ as you want:
  71   *  this means you may launch jobs using various GrUI nodes simultaneously
  72   *  if you so wish. And even split the jobs througho various, separate 
  73   *  Grids if you feel like it, hence potentially increasing your 
  74   *  throughput and computing power, harnessing even more resources.
  75   *
  76   *  Detaching your jobs from the GrUI has one serious drawback though:
  77   *  all your job information must travel from your service user interface
  78   *  to the grid user interface through the Internet, which may be
  79   *  potentially dangerous. We deal with this security issue using SSH
  80   *  to handle all communications and provide encryption. As long as 
  81   *  you use strong passwords you may feel secure. Actually, it is as
  82   *  weak (or strong) as working directly on the GrUI node: access to
  83   *  it is still managed by standard password mechanisms and subject to
  84   *  the same types of attacks. However, remember you now have an extra
  85   *  system (the front-end) to maintain and secure. Please, be always
  86   *  cautious with any server you use.
  87   *
  88   *  Sounds convincing? Then read ahead to learn how to use this class.
  89   *
  90   *  <b>PERSISTENT vs. DISCONNECTED MODE</b>
  91   *
  92   *  As of release 2.0, this distinction is no longer meaningful. We now
  93   * use a new SExec class that multiplexes disconnected commands over a
  94   * single sahred channel, thus providing all the advantages of both,
  95   * connected and disconnected modes. We now provide a single set of
  96   * methods.
  97   *
  98   *  <b>JOB MANAGEMENT</b>
  99   *
 100   *  In order to submit your jobs to the grid you need to understand how
 101   *  job management has been defined for this class. On the command line
 102   *  you would have a lot more versatility, but to make this class more
 103   *  useful some compromises had to be reached. We have defined a strict
 104   *  protocol to generate/prepare your jobs before submitting them to
 105   *  the grid, and you must stick to it if you want to avoid problems.
 106   *
 107   *  To understand why this has been designed the way it is, you should
 108   *  keep in mind that you will be preparing your jobs on one (or many)
 109   *  front-end and submitting them from it to one (or many) GrUI nodes.
 110   *  Further to it, this has been designed to make it easy to deploy
 111   *  web-based services for users. Therefore many similar jobs might
 112   *  be launched simultaneously and we need some way to avoid collusion
 113   *  among them. To avoid one job stepping over other we must isolate
 114   *  every one from each other. This means providing an isolated environment
 115   *  for every job.
 116   *
 117   *  The easiest way to achieve our goal is to have every job submitted
 118   *  placed on an independent directory (which we pair to the job name). 
 119   *  For single jobs, this means that you should make sure that no two 
 120   *  potentially simultaneous/overlapping jobs have the same name (i.e. 
 121   *  are stored in the same directory so there is no risk one overwrites
 122   *  files of the other).
 123   *
 124   *  Sometimes this may result inconvenient to you. E.g. if your whole job
 125   *  is submitted split into many separate sub-jobs (which each is a separate
 126   *  job from the point of view of the grid) you may want to follow some
 127   *  naming convention for your sub-jobs that makes it easier to identify
 128   *  and keep track of them. In this case, if you had two simultaneous
 129   *  runs, then the names would collide.
 130   *
 131   *  For example, let's say you are rendering frames of a movie and want
 132   *  to identify each job by frame number: 0000, 0001, 0002, 0003... If
 133   *  you now try to generate a second movie while the first is being
 134   *  processed, then the frames of the second movie (named as well 0000,
 135   *  0001, etc...) would overwrite the frames of the first one.
 136   *  Generating random names for each frame would be an option, but too
 137   *  cumbersome and expensive as you would need to keep track of the
 138   *  association of the random names with the actual frames.
 139   *
 140   *  To deal with this scenario we define 'sessions'. A session is identified
 141   *  by a unique identifier, and guarantees that all jobs belonging to this
 142   *  session are kept separate from similarly named jobs from other sessions.
 143   *
 144   *  Actually when you create a session, what we actually do is create
 145   *  a subdirectory in the GrUI and direct all further jobs to this 
 146   *  subdirectory. This way, jobs of two sessions may have the same name
 147   *  and not step into each other.
 148   *
 149   *  <b>PREPARING JOBS FOR THE GRID</b>
 150   *
 151   *  To prepare a job for the grid you must assign it a name. The same
 152   *  preacutions that apply to any local job hold for your grid work too:
 153   *  if various simultaneous jobs of the same kind may be run, then each
 154   *  must be kept separate from the others by using a different name.
 155   *
 156   *  Once you have decided the name, you must create a directory locally
 157   *  with the same name as your job. In this directory you must install
 158   *  everything needed to run your job: executables, libraries, input
 159   *  data and a JDL file.
 160   *
 161   *  The JDL file defines the work that we will ask the grid to carry
 162   *  out. Since each single job gets its own directory, you will only
 163   *  submit one JDL file for each, and to make processing easier, we
 164   *  request that this JDL file have a fixed name: "job.jdl".
 165   *
 166   *  The grid processing will generate various auxiliary files, for internal
 167   *  housekeeping. Again, for simplicity, we have chosen to call each of
 168   *  them 'job.*', i.e. 'job.' something. This means that other than 'job.jdl'
 169   *  you should NOT create any file named job.anything on your job directory
 170   *  to avoid collusion with possible temporary files.
 171   *
 172   *  In brief, to prepare a job:
 173   *  - select a name
 174   *  - create a directory named after the job
 175   *  - populate this directory with all files needed to run your job
 176   *  - generate the file 'job.jdl' with the description of the work to
 177   *    be carried out by the grid
 178   *  - avoid having files named 'job.*' (starting with 'job.')
 179   *
 180   *  <b>SUBMITTING JOBS TO THE GRID</b>
 181   *
 182   *  First consider whether you will be using unique job names or if
 183   *  you will follow a convenient naming convention that may cause name
 184   *  collisions with other jobs.
 185   *
 186   *  If you feel pretty safe that the job name is unique (e.g. has been
 187   *  generated using one or more random strings), then simply call the
 188   *  appropriate *job_submit() function.
 189   *
 190   *  If you are using names that have low entropy or reusing names for
 191   *  similar jobs then it is advisable that you first call one of the
 192   *  *session_new() routines to ensure all your jobs will be kept isolated
 193   *  from other similarly named jobs, and then use the *job_submit()
 194   *  routines to send your jobs.
 195   *
 196   *  For the curious: when you submit your job, the directory and all
 197   *  of its contents will be sent to the remote Grid UI selected and
 198   *  then the 'job.jdl' will be submitted to the grid. In the process,
 199   *  several files will be generated holding information about your
 200   *  job identity in the grid context that will be kept for housekeeping
 201   *  and future reference.
 202   *
 203   *  The above is to be kept in mind when submitting light or numerous
 204   *  jobs: the transfer time may become sensibly relevant. Please do 
 205   *  take it into consideration in your equations when designing jobs
 206   *  for the grid using this class. You may find it interesting to first
 207   *  store all or some of your job data/execs on the grid and keep them
 208   *  already there instead of having to copy them.
 209   *
 210   *  Grid file management routines are not included yet, but are intended
 211   *  for a future release of this class.

 325      /**
 326       *    Submit timeout
 327       *    XXX JR XXX HACK
 328       *    This variable used for a hack (?) to ensure job submission does
 329       * not hang. Currently job submission using edg-job-submit may hang
 330       * occassinally. This is rare, maybe one out of 4-5 thousand jobs,
 331       * but for such large datasets it becomes an issue.
 332       *
 333       *    My impression is that it hangs on copying the job to the remote
 334       * resource broker. This may happen if the RB becomes unavaliable:
 335       * by TCP semantics the line remains open until connectivity is
 336       * restored, but if this takes too long, or if some error conditions
 337       * are met your connection may stall.
 338       *
 339       *    This is very difficult to spot unless it is done at a very low
 340       * level in the copy connection protocol. Higher up, one can not know
 341       * if the copy takes too long because of connection problems or if it
 342       * simply is a huge dataset or a very slow connection.
 343       *
 344       *    For the same reason, edg-job-submit has no way to know if the
 345       * connection is stalled unless it has some way to measure the amount
 346       * of data transferred (which is done lower in the protocol stack). So
 347       * from edg-job-submit point of view, most probably it is not a bug,
 348       * but a feature: it aims for a most optimistic situation and a most
 349       * safe transfer, i.e. keep trying to use the connection as long as it
 350       * is not broken, and try to ensure the data gets fully transferred
 351       * in the presence of trouble for as long as the connection may be
 352       * recovered, and do not make any assumption about the size of the
 353       * dataset or the speed of the line.
 354       *
 355       *    Now, for an automated submission system with deadline (proxy
 356       * validity) this is a potential disaster: I have witnessed job submissions
 357       * hang for as long as a couple of days (and then I killed them manually).
 358       *
 359       *  So, what can we do?
 360       *
 361       *    If you know the bounds of your job sizes (exec + datasets) then you
 362       * may have an estimate of how long you want to allow a transfer to 
 363       * take, and kill any submission taking longer. Then with appropriate
 364       * code the submission may be retaken (in my experience, hang submissions
 365       * usually succeed after bein manually killed and retried) or the job
 366       * resubmitted at a later time if you have recovery code on your app.
 367       *
 368       *    This variable holds the maximum time in seconds you guesstimate a 
 369       * job submission should be allowed to take. If this timeout is 
 370       * exceeded the submission will be unpiteously killed. Since this is
 371       * a rare condition you may be generous.
 372       *
 373       * @access private
 374       * @var integer
 375       */

 745      /**
 746       * open a persistent connection to the Grid UI server
 747       * 
 748       *     The Grid User Interface Server is the entry point to the Grid
 749       *  for users and user applications. This is where jobs are launched from.
 750       * 
 751       *     This package has been designed to be able to be installed in any
 752       *  host, independent of whether it is an UI or not. Thus, to be able to
 753       *  submit jobs to the Grid, the server hosting the Web UI must connect to
 754       *  a Grid UI host to do the work.
 755       * 
 756       *     This routine opens a connection to a Grid UI host using an specified
 757       *  username (i.e. all jobs will be run under said username).
 758       * 
 759       *     The panorama therefore will look like this:
 760       * 
 761       *     HTML front-end --> processor.php <--> SSH <--> remote host <--> Grid
 762       *
 763       *    This allows for better resilience: should a GridUI host be 
 764       * unavailable, we can detect the error condition and try another 
 765       * one. If the GridUI runs the front-end, then we have a single
 766       * point of failure, which is a no-no.
 767       *
 768       * Sample usage:
 769       * <code>
 770       * require_once './grid_config.php';
 771       * require_once './ssh.php';
 772       * require_once './grid.php';
 773       * 
 774       * $eg = new Grid;
 775       * 
 776       * $eg->pconnect();
 777       * if ($eg->get_connection_status() == FALSE)
 778       *     echo "Couldn't instantiate a new Grid!\n";
 779       * </code>
 780       *
 781       *    @note    Use files instead of pipes and open them after securing
 782       * thew connection: this should do with deadlocks and leave a trace
 783       * log.
 784       *
 785       *    @return TRUE on success, FALSE otherwise.
 786       */

 789      {

 791

 795      // object creates a master channel to the remote end point
 796      // The master channel will be hitchhicked by all other
 797      // SExec commands.
 798      
 799      $this->sx = new SExec($this->entry_point, $this->password);

 808      /**
 809       * Start the Grid services
 810       *
 811       *    This function starts the grid services on the remote UI server host.
 812       *    This is done by unlocking the certificate that we are going to use 
 813       *  to run our jobs on the grid using the passphrase provided.
 814       *
 815       *    Grid services have a lifetime of their own. By default they are
 816       *    available for 12:00 hours (that's the default value of
 817       *    grid-proxy-init itself), but their duration may be fine tuned
 818       *    if we have some knowledge about the time required to run our
 819       *    job.
 820       *
 821       *    Grid opening time is specified in hours+minutes. If the number of
 822       *    minutes is negative, the specified minutes are substracted from
 823       *    the specified hours (e.g: 1, -15 is fifteen minutes to one hour,
 824       *    i.e. 45 minutes). If the total time specified is negative then
 825       *    the default of 12:00 is used.
 826       *
 827       *    Grid::initialize() enables the grid for a specified amount of time
 828       * (by default 12:00h). This means that during the validity period, 
 829       * the user on the Grid-UI host may access the grid, in the same or
 830       * different logins. The validity period SURVIVES after we close 
 831       * all communications with the remote grid entry point for as long
 832       * as we have specified (so our jobs may continue running).
 833       *
 834       *    If Grid access was already available (by a previous call to 
 835       * Grid::initialize()) when we issue the call, then it is reused
 836       * and extended to acommodate the newly requested validity period.
 837       * In other words, the Grid access is shared among all logins
 838       * during its lifetime.
 839       *
 840       *    This also means that if a valid certificate has been issued and
 841       * not expired yet (another call to Grid::initialize() is still
 842       * valid), then we may submit jobs to the Grid without any need
 843       * to call Grid::initialize() ourselves.
 844       *
 845       *    E.g., say you have a web-based service that runs a long job
 846       * and you want to have the grid enabled 12h (the default). You
 847       * just call Grid::initialize() and then submit the job.
 848       *
 849       *    Now say that before it expires, someone logs in on your account 
 850       * but shouldn't have access to the Grid (i.e. they don't know
 851       * the Grid-activation passphrase). Since the grid is already
 852       * activated, they CAN submit jobs on your behalf even if they
 853       * should not.
 854       *
 855       *    Therefore, <b>DO NOT SHARE YOUR ACCOUNT ON THE GRID-UI
 856       * WITH ANYBODY</b>. Protect it as dearly as your Grid certificate.
 857       *
 858       *    It also means that debugging may be somewhat convoluted, as
 859       * a call to Grid::initialize() may fail and jobs could still
 860       * be accepted if another call from some other process is still
 861       * valid. While debugging, it is better if you review the 
 862       * command output and make sure it shows how the call fared.
 863       *
 864       *    So, what happens if we destroy a session? See Grid::destroy()
 865       * for more details.
 866       *
 867       * Sample usage:
 868       * <code>
 869       *     $eg = new Grid;
 870       *     $eg->set_user($user);
 871       *     $eg->set_host($host);
 872       *     $eg->set_password($passwd);
 873       *     $eg->set_passphrase($passphrase);
 874       *     $eg->set_work_dir("/tmp/grid/test/cless");
 875       *     $eg->set_error_log("/tmp/grid/test/cless/connection.err");
 876       *     $eg->connect();
 877       *     if (!$eg->initialize())
 878       *         echo "error: couldn't init the grid\n";
 879       *     else
 880       *         echo "OK\n";
 881       *     $eg->destroy();
 882       *     $eg->disconnect();
 883       * </code>
 884       *
 885       *    @note the output of the grid initialization command will go to
 886       *          <b>our</b> standard output (i.e. the web server)!
 887       *
 888       *    @param    integer        Estimated duration in hours of the session
 889       *
 890       *    @param    integer     Estimated duration in minutes of the session
 891       *
 892       *    @return bool TRUE on success, FALSE otherwise
 893       */
 894       
 895      function initialize($hours=12, $minutes=0)
 896      {
 897          global $debug_grid;
 898      
 899      if ($debug_grid) echo "Grid::initialize($hours, $minutes)\n";
 900      
 901      // connect if not connected
 902       if ($this->connected == FALSE)
 903          if ($this->connect() == FALSE)
 904          return FALSE;
 905      
 906      // validate and compute activation time to use
 907          $total_minutes = ($hours * 60) + ($minutes);
 908      if ($total_minutes < 0) {
 909          //use default
 910          $hours = 12;
 911          $minutes = 0;
 912      } else {
 913          $hours = floor($total_minutes / 60);
 914          $minutes = $total_minutes % 60;
 915      }
 916      
 917      $out = "";
 918      // Create remote working dir
 919      $status = $this->sx->ssh_exec(
 920          "mkdir -p $this->work_dir/default",
 921          $out);
 922      if ($status != 0) {
 923          // something went airy
 924          if ($debug_grid) {
 925              echo "--> mkdir -p $this->work_dir/default\n";
 926              echo "--> exit: $status\n";
 927              echo "--> output:\n".$out . "\n\n";
 928          }
 929          return FALSE;
 930      }
 931      $this->sessions[] = 'default';        // enter on sessions list
 932      
 933      // NOTE: XXX JR XXX
 934      //  Using popen here has one major drawback: the command output
 935      //  will go directly to the web page!
 936      //  We should consider instead getting stdin/stdout handles or
 937      //  dismissing remote command output.
 938          $rin = $this->sx->ssh_popen(
 939          ". /etc/bashrc ; " .
 940          "/opt/globus/bin/grid-proxy-init -pwstdin -valid $hours:$minutes", 
 941          "w"
 942          );
 943      sleep(1);   // leave some time for the prompt (and preceding flush)
 944      if ($debug_grid) echo "Remote INput = $rin\n";
 945      fputs($rin, $this->passphrase."\n"); fflush($rin);
 946      $status = $this->sx->ssh_pclose($rin);
 947      
 948      if ($status == 0) {
 949          return $this->initialized = TRUE;
 950      } else {
 951          if ($debug_grid) echo "PCLOSE STATUS = $status\n";
 952          return $this->initialized = FALSE;
 953      }
 954      }

 956      /**
 957       *    Destroy remote grid identity
 958       *
 959       *    Destroy the certification we initialized so that no more jobs
 960       *    can be launched under our identity.
 961       *
 962       *    This may be called even if we haven't called 'Grid::initialize()'
 963       * because there may exits a previous activation that is still 
 964       * valid and we want to destroy it.
 965       *
 966       *    To make things clear:
 967       *
 968       *    Grid::initialize() "opens" the "door" to the Grid for <i>the user</i>
 969       * during a given time. New calls from the same user in this or any
 970       * other login session, from this or any other site, while the "door" 
 971       * is open, share the same "door" and simply extend its validity
 972       * period.
 973       *
 974       *    Grid::destroy() "closes" the currently open door. If the door
 975       * was being shared by more login sessions, it is closed for <i>all</i>
 976       * of them, not just the caller, and hence nor the caller, nor
 977       * <i>any process</i> under the same user will be able to use the
 978       * grid any longer unless Grid::initialize() is called again to
 979       * open the door again (issue a new certificate).
 980       *
 981       *    In other words, you don't close <i>a</i> Grid "door", you close 
 982       * <i>the</i> Grid "door", and if it casually is being shared with other 
 983       * work sessions, then ALL of them will be destroyed (meaning that 
 984       * other active work sessions will fail).
 985       *
 986       *    Thus: be careful when using this method. Be <b>very careful</b>.
 987       *
 988       *    Sessions should be initiated using a validity length that you
 989       * guesstimate will be comfortably enough for running all your work
 990       * and left to expire by themselves.
 991       *
 992       *    Grid::destroy() should only be called when you are sure that
 993       * you don't want <i>any</i> work on the Grid to be accepted on
 994       * your behalf, neither from this not other work sessions.
 995       *
 996       *    So, in general, it is better to make good estimations of the
 997       * time needed by your jobs and specify it to Grid::initialize()
 998       * and not use Grid::destroy() unless there are good reasons.
 999       *
1000       * Sample usage:
1001       * <code>
1002       *     $eg = new Grid;
1003       *     $eg->set_user($user);
1004       *     $eg->set_host($host);
1005       *     $eg->set_password($passwd);
1006       *     $eg->set_passphrase($passphrase);
1007       *     $eg->set_work_dir("/tmp/grid/test/cless");
1008       *     $eg->set_error_log("/tmp/grid/test/cless/connection.err");
1009       *     if (!$eg->initialize())
1010       *         echo "error: couldn't init the grid\n";
1011       *     else
1012       *         echo "OK";
1013       *     $eg->destroy();
1014       *     $eg->disconnect();
1015       * </code>
1016       *
1017       *    @note    Be careful when using this function: as it destroys our
1018       *            Grid-ID, no more work will be able to be executed on the
1019       *            grid on our behalf. In other words, please, make sure there
1020       *            is no work pending and that all your work has terminated
1021       *            before destroying your Grid-ID.
1022       *
1023       *    @return bool TRUE on success, FALSE on failure
1024       */
1025      function destroy()
1026      {
1027          global $debug_grid;
1028      
1029          if ($debug_grid) echo "Grid::destroy()\n";
1030      
1031      if (! $this->connected)
1032              if ($this->connect() == FALSE)
1033          return FALSE;
1034  
1035          $out = "";
1036          $ret = $this->sx->ssh_exec(
1037          ". /etc/bashrc ; grid-proxy-destroy",
1038          $out);
1039      if ($ret == 0)
1040          return TRUE;    // OK
1041      else {
1042          if ($debug_grid) {
1043              echo "--> exit: $ret\n";
1044              echo "--> output:\n".$out . "\n\n";
1045          }
1046          return FALSE;
1047      }
1048      }

1050      /**
1051       *    Create a new session
1052       *
1053       *    We take control of the session system: in a previous incarnation
1054       * we got the session name from the user. This is dangerous since we
1055       * can not avoid clashes even with random session names provided by
1056       * a user: a randomly generated name may be guaranteed unique within
1057       * the user's local host namespace, but should various calls come from
1058       * different hosts (e.g. HA front-ends in a cluster), it is conceivable
1059       * (although improbable) that both come up with the same random number
1060       * and generate a clash on the shared remote grid access point. Hence 
1061       * the new approach: to be true, users do not need to generate a session 
1062       * name themselves: they just need a way to refer to them.
1063       *
1064       *    This routine will generate a new session: under a given session,
1065       * we guarantee that processes will be run under an isolated sandbox
1066       * where non name clashes from other concurrent users will occur.
1067       * Name clashes within a given session induced by the user are still
1068       * the responsability of the user.
1069       *
1070       * SESSIONS
1071       *
1072       *    The basic idea is as follows: if you are developing a service that
1073       * may be called concurrently by various users, your problem at the GrUI
1074       * is the same as on your local server: avoiding name clashes for jobs.
1075       * As long as you manage that locally using unique names, the same will
1076       * work on the UI server.
1077       *
1078       *    This however may be inconvenient at times. It is usually the case
1079       * when your service is not composed by a single job, but by many
1080       * independent jobs that may be run concurrently as well. You still 
1081       * need to generate unique names for each sub-job, but if they are
1082       * produced within a local uniquely named service-instance, then you
1083       * may reuse the sub-job names for each instance.
1084       *
1085       *    This comes handy in the case of many sub-jobs: as long as the main
1086       * service instance is uniquely identified, contained sub-jobs may have
1087       * significant names that are easier to identify than randomly-generated
1088       * ones: e.g.
1089       * <pre>
1090       *    user A -> service-instance-A -+-> sub-job-1
1091       *                                  |
1092       *                                  +-> sub-job-2
1093       *                                  |
1094       *                                  etc...
1095       *
1096       *    user B -> service-instance-B -+-> sub-job-1
1097       *                                  |
1098       *                                  +-> sub-job-2
1099       *                                  |
1100       *                                  etc...
1101       * </pre>
1102       *    In this case, since the service-instances have unique names (A, B)
1103       * we may use the same naming strategy in both cases for naming sub-jobs
1104       * (1, 2,...) which makes bookkeeping a lot easier.
1105       *
1106       *    To reproduce a similar behaviour remotely we provide 'sessions'.
1107       *
1108       *    Basically, what you are doing in the local case is isolating all the
1109       * equally named jobs of each service-instance within an unique sandbox.
1110       * In the grid you get the same result using 'sessions': whenever you
1111       * want to submit a series of non-randomly-named jobs (or even a single
1112       * one) you first allocate a session, and then attach those jobs to the
1113       * session. Job names within a session are guaranteed not to clash with
1114       * equal job names from another session.
1115       *
1116       *    Note the 'non-randomly-named' tag above: you want to use sessions
1117       * ALWAYS that you use any non-random job name unless you can guarantee
1118       * it will be the only job with that name ON THE GRID.
1119       *
1120       *    This is an important notice: your job may have a non-random, but
1121       * guaranteed unique name on your local host. And as long as the job
1122       * will only be submitted from your local host it is OK. But if you
1123       * are going to share your tools with other fellows, then they will
1124       * install them locally and submit a similarly named job as well. The
1125       * job name will be unique within each one's local machine, but when
1126       * jobs are collected at the GrUI, they will all have the same name,
1127       * which is a no-no.
1128       *
1129       * USING SESSIONS
1130       *
1131       *    Once you see the need for using sessions, using them is quite 
1132       * simple: you call this function asking for a new session to be
1133       * created on the remote grid access point on your behalf. The routine
1134       * will return a unique session identifier which you later use to 
1135       * tag jobs that are to be run within that session's sandbox.
1136       *
1137       * Sample usage:
1138       * <code>
1139       *    $eg = new Grid;
1140       *    $eg->set_user($user);
1141       *    $eg->set_host($host);
1142       *    $eg->set_password($passwd);
1143       *    $eg->set_passphrase($passphrase);
1144       *    $eg->set_work_dir("/tmp/grid/debug");
1145       *    $eg->set_error_log("/tmp/grid/debug/connection.err");
1146       *    echo "initializing grid... ";
1147       *    if (!$eg->initialize()) {
1148       *        echo "error: couldn't init the grid\n";
1149       *        exit;
1150       *  }
1151       *    else
1152       *        echo "OK\n";
1153       *    echo "Submitting tst-job to default session...\n";
1154       *    $out = array("");
1155       *    if (! $eg->job_submit("tst-job", $out)) {
1156       *        echo "error: coudn't start the job\n";
1157       *        exit;
1158       *    }
1159       *    else 
1160       *       echo "OK\n";
1161       *    print_r($out);
1162       *    echo "Submitting tst-job to a new session...\n";
1163       *    $sess = $eg->session_new();
1164       *    echo "sess = "; print_r($sess); echo "\n";
1165       *    if ($sess != FALSE) {
1166       *        $out = array("");
1167       *        if (! $eg->job_submit("tst-job", $out, $sess))
1168       *           echo "error: coudn't start the job\n";
1169       *        else {
1170       *           echo "OK\n";
1171       *        }
1172       *        $eg->session_destroy($sess);
1173       *    }
1174       *    $eg->destruct();
1175       * </code>
1176       *
1177       * @param string $hint an optional string to be used for the session name
1178       *
1179       * @return string|false session ID of the newly generated session.
1180       */
1181      function session_new($hint="sess")
1182      {
1183          global $debug_grid;
1184      
1185      if ($debug_grid) echo "Grid::session_new()\n";
1186      
1187      // we need a valid connection, no need for initialization.
1188      if (! $this->connected)
1189          if (! $this->connect())
1190              return FALSE;
1191      
1192      $out=array("");
1193      $status = $this->sx->ssh_exec( 
1194              "mktemp -d $this->work_dir/$hint.XXXXXX",
1195              $out);
1196          if ($status == 0) {
1197          if ($debug_grid) print_r($out);

1202              // use the generated one

1205          $this->sessions[$hint] = $sess;
1206          return $hint;
1207      } else {

1209          return FALSE;

1402      /**

1426       *     $eg->set_passphrase($passphrase);
1427       *     $eg->set_work_dir("/tmp/grid/test/cless");
1428       *     $eg->set_error_log("/tmp/grid/test/cless/connection.err");
1429       *     echo "initializing grid... ";

1431       *         echo "error: couldn't init the grid\n";
1432       *     else
1433       *         echo "OK\n";

1435       *     if (! $eg->job_submit("tst-job", $out))
1436       *         echo "error: coudn't start the job\n";
1437       *     else 
1438       *         echo "OK\n";

1442       * </code>
1443       *
1444       *    @note in future instances we may provide routines to generate the 
1445       *        JDL, possibly within a GridJob class of its own.
1446       *
1447       *    @param    string    The name of the job (same as the subdirectory it is in)
1448       *    @param array      A variable to hold any messages spitted by the submission
1449       *                    procedure. Messages will be stored as an array of
1450       *                    strings (one per line) without ending newlines.
1451       *    @param    string Optional name of the session to which this job belongs
1452       *                    (obtained from a previous call to session_new).
1453       *  @return bool TRUE if success, FALSE otherwise
1454       */
1455      function job_submit($job, &$out, $session='default')
1456      {
1457          global $debug_grid;
1458      
1459      if ($debug_grid) {
1460          echo "Grid::job_submit($job, $out, $session)\n";
1461          echo "Copying job $job to remote end ".
1462              "$this->entry_point:$this->work_dir/$session\n";
1463      }
1464      
1465      // check this first as it is cheaper
1466      if (! $this->session_is_valid($session))
1467          return FALSE;
1468      $sd = $this->sessions[$session];
1469      
1470      // we need an initialized grid
1471      if (! $this->initialized)
1472          if (! $this->initialize())
1473              return FALSE;
1474      
1475      // We should check for $job validity (i.e., $job/job.jdl exists)
1476      
1477          // Copy job details over to remote end
1478      $status = $this->sx->ssh_copy_to(
1479          $job, "$this->work_dir/$sd", $out);
1480      if ($status == FALSE) {
1481          // copy failed
1482          if ($debug_grid) {
1483              echo "--> copy: failed\n";
1484          print_r($out);
1485          }
1486          return FALSE;
1487      }
1488      // XXX JR XXX
1489      // Remove any existing 'identifier.txt' file
1490      //  This is 'cos otherwise, if we are re-invoked to re-submit
1491      // the job, the new ID will be appended and job_get_id will
1492      // return the old, failed one.
1493      //  Ideally we should have a job_resubmit() routine that takes
1494      // care of it AND keeps track of the number of resubmissions...
1495      $this->sx->ssh_exec("rm -f $this->work_dir/$sd/$job/identifier.txt",
1496          $out);
1497      unset($out);    // we ignore output and exit code!
1498      
1499      if ($debug_grid)
1500          echo "Submitting with a timeout of $this->submit_timeout:\n\t"
1501          . (($this->submit_timeout != 0)? 
1502               "(sleep 10; pkill -P \$\$ ) & " :
1503           " ")
1504          . "/opt/edg/bin/edg-job-submit"
1505          . "  -o $this->work_dir/$sd/$job/identifier.txt"
1506          . "  $this->work_dir/$sd/$job/job.jdl"
1507          . "  | tee $this->work_dir/$sd/$job/submit.txt\n";
1508  
1509          // NOTE: with direct execution we don't go through the login
1510      // process, hence needed environment variables must be set.
1511          // XXX JR XXX
1512      //      There is an alternate solution: use the remote
1513      //      command "sh --login -c $command"
1514      //      This tells sh to behave as a login shell, and read
1515      //      commands from the string $command.
1516      $status = $this->sx->ssh_exec(
1517          "("
1518          . ". /etc/bashrc ;"
1519          . "export EDG_WL_LOCATION=/opt/edg ;"
1520          . "export EDG_WL_LIBRARY_PATH=/opt/globus/lib:/opt/edg/lib ;"
1521          . "cd $this->work_dir/$sd/$job; "
1522          . (($this->submit_timeout != 0) ?
1523               "(sleep $this->submit_timeout; pkill -P \$\$ ) & " :
1524           " ")
1525              . "/opt/edg/bin/edg-job-submit"
1526          . "  -o $this->work_dir/$sd/$job/identifier.txt"
1527          . "  $this->work_dir/$sd/$job/job.jdl"
1528          . "  | tee $this->work_dir/$sd/$job/submit.txt"
1529          . ")",
1530          $out);
1531      if ($debug_grid) echo "$session::$job ($sd/$job) submit status was $status\n";
1532      if ($status == 0) {
1533          // at this point we may as well copy identifier.txt and submit.txt
1534          // to the local directory and have them cached here.
1535          $this->sx->ssh_copy_from("$this->work_dir/$sd/$job/submit.txt",
1536              $job, $out);
1537          // note that identifier.txt may not exist (it SHOULD though!)
1538          $this->sx->ssh_copy_from("$this->work_dir/$sd/$job/identifier.txt",
1539              $job, $out);
1540          return TRUE;
1541      }
1542      else
1543          return FALSE;
1544      }
